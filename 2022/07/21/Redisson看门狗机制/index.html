Redisson看门狗机制

发表于 2022-07-21 | 分类于 Lock

看门狗（Watchdog）机制是 Redisson 为分布式锁提供的一种自动续期功能。它能够在客户端持有锁期间，自动延长分布式锁的有效期，以防止因任务执行时间过长、而导致锁提前过期被其他客户端获取，从而破坏分布式锁的互斥性。
看门狗机制特别适用于任务执行时间较长、但又不确定的场景，确保锁在客户端存活时不会意外失效。它通过后台定时任务实现续期，避免了开发者手动管理锁超时的复杂性。

01 工作原理

Redisson的看门狗机制通过定时任务和 Redis 的 TTL 管理以实现锁的自动续期，如下图所示：

主要工作流程如下：

1. 锁的获取与看门狗初始化

当客户端调用 lock() 方法且未指定过期时间（leaseTime）时：

默认有效期：Redisson 为锁设置初始 TTL，默认值为 30,000 毫秒（30 秒）。锁记录存储在 Redis 中，使用哈希结构（HSET）保存锁名称、所有者标识（UUID + 线程 ID）及重入计数。

看门狗触发：锁获取成功后，若未指定 leaseTime，Redisson 调用 scheduleExpirationRenewal 初始化看门狗任务。

2. 定时任务的启动与续期间隔

续期间隔：看门狗默认为“每隔 TTL/ 3（30,000 / 3 = 10,000 毫秒，即 10 秒）”检查一次锁状态。这是通过 ConnectionManager 的 newTimeout 方法调度实现的。

任务调度：定时任务基于 Netty 的 HashedWheelTimer，首次延迟 10 秒后执行，之后根据续期结果递归调度。

3. 锁的续期过程

每次任务触发时：

状态检查：从本地 EXPIRATION_RENEWAL_MAP 获取锁的 ExpirationEntry，确认锁是否仍由当前客户端持有。

续期命令：通过 renewExpirationAsync 发送 Lua 脚本到 Redis：

if (redis.call(‘hexists’, KEYS[1], ARGV[2]) == 1) then
redis.call(‘pexpire’, KEYS[1], ARGV[1]);
return 1;
end;
return 0;
检查锁的所有者是否匹配，若匹配则更新 TTL 为 30 秒。

异步执行：续期操作异步完成，成功后调度下一次续期。

4. 续期的循环机制

只要锁未被释放、且客户端与 Redis 连接正常，则续期任务持续运行。如下所示：

T=0 秒：锁获取，TTL=30 秒。
T=10 秒：续期，TTL 重置为 30 秒。
T=20 秒：续期，TTL 重置为 30 秒。
……以此类推。

5. 锁的释放与看门狗停止

主动释放：调用 unlock() 删除 Redis 锁记录，并清理本地续期任务。

客户端崩溃：若客户端断连，续期停止，锁在最后一次续期后的 30 秒内释放。

6. 其他事项

网络延迟：续期请求延迟可能导致锁短暂过期，Redisson 通过异步重试缓解。

配置影响：TTL 是看门狗机制中锁的默认有效期，其值的大小会直接影响系统行为：

如果设置得太短（例如几秒钟），看门狗需要更频繁地执行续期操作（ TTL / 3 ），这会增加客户端和 Redis 服务器的负载，影响系统的性能。

如果设置得太长（例如几分钟），续期间隔也会相应变长（例如每分钟检查一次），可能导致锁在客户端意外中断后无法及时释放，影响系统的响应速度和资源利用效率。
