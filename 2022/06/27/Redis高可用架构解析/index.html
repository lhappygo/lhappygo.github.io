Redis高可用架构解析

发表于 2022-06-27 | 分类于 Redis

Redis 高可用架构最常用、最成熟的方式是 哨兵模式（Sentinel）。它是在主从复制基础上发展而来，专门为了解决主节点故障自动切换问题的官方解决方案。

以下是对哨兵模式的详细解读：

一、 基础：主从复制（Replication）

哨兵模式建立在主从复制之上，理解主从复制是前提：

  1. 核心结构：
    • 主节点（Master）： 负责处理写操作（SET, DEL, INCR等）。数据变更会异步复制到从节点。
    • 从节点（Slave/Replica）： 连接主节点，接收并应用主节点发送的数据变更流（replication stream）。主要处理读操作（GET, LRANGE, HGETALL等），也可以配置为只读。一个主节点可以有多个从节点。
  1. 优点：
    • 数据冗余： 数据在多个节点上有备份，提高数据安全性。
    • 读扩展： 通过将读请求分发到多个从节点，可以显著提高系统的读吞吐量。
    • 基础故障恢复： 如果从节点故障，影响较小（除非所有从节点都故障）。如果主节点故障… 问题来了：需要人工干预！
  2. 缺点（高可用视角）：
    • 主节点单点故障： 主节点故障意味着整个 Redis 服务无法写入。虽然有从节点持有数据，但需要手动执行SLAVEOF no one将一个从节点提升为主节点，并重新配置其他从节点和客户端指向新的主节点。这个过程慢、易出错、服务中断时间长。

二、 哨兵模式（Sentinel）：解决主节点自动故障转移

哨兵系统是一个分布式系统，通常由奇数个（3个或5个最常见）Sentinel 进程/节点组成。它不存储数据，只负责监控 Redis 节点（主节点和从节点）的状态，并在主节点故障时自动执行故障转移操作。

核心功能

  1. 监控（Monitoring）：
    • Sentinel 节点会持续检查主节点和从节点的健康状态（是否可达、是否正常运行）。
    • 通过定期向所有被监控的节点发送 PING 命令并等待响应来判断节点状态。
  2. 通知（Notification）：
    • 当某个被监控的 Redis 节点发生状态变化（如主观下线、客观下线、故障转移开始/结束、新主节点选出等）时，Sentinel 可以通过配置的 API 或脚本通知系统管理员或其他应用程序。
  3. 自动故障转移（Automatic Failover）：
    • 主观下线（SDOWN - Subjectively Down）： 当一个 Sentinel 节点在配置的超时时间内（down-after-milliseconds）没有收到主节点的有效 PING 响应，它主观地认为该主节点下线了。
    • 客观下线（ODOWN - Objectively Down）：
      • 当一个 Sentinel 认为主节点主观下线后，它会询问集群中其他 Sentinel 节点对该主节点状态的判断。
      • 如果达到配置的法定人数（quorum） (通常是 Sentinel节点数/2 + 1) 的 Sentinel 节点都报告该主节点主观下线，那么这个主节点就被标记为客观下线。
      • 这是触发自动故障转移的关键条件，避免了单个 Sentinel 误判导致不必要的切换。
    • 选举领导者 Sentinel（Leader Election）：
      • 一旦主节点被判定为客观下线，所有 Sentinel 节点会通过 Raft 算法选举出一个领导者 Sentinel 来负责执行本次故障转移操作。选举需要获得大多数（majority） Sentinel 的同意（通常是 Sentinel节点数/2 + 1）。
    • 选择新主节点（Promotion）：
      • 领导者 Sentinel 会从原主节点的从节点列表中，根据一定的规则（优先级slave-priority、复制偏移量replication offset、运行IDrun id等，优先选择复制最完整、优先级高的）选出一个最合适的从节点。
      • 领导者 Sentinel 向选中的从节点发送 SLAVEOF no one 命令，将其提升为新的主节点。
    • 重新配置从节点（Reconfiguration）：
      • 领导者 Sentinel 向其他所有从节点发送新的 SLAVEOF 命令，让它们改为复制新的主节点。
    • 更新配置（Configuration Propagation）：
      • 领导者 Sentinel 将新的主节点信息广播给集群中的其他 Sentinel 节点。其他 Sentinel 节点会更新其内部配置。
      • 如果原主节点恢复（例如网络问题解决了），领导者 Sentinel 会向其发送 SLAVEOF 命令，让它成为新主节点的从节点。
  4. 配置提供者（Configuration Provider）：
    • 客户端应用程序在启动时，不是直接连接主节点，而是连接 Sentinel 节点。
    • 客户端通过向 Sentinel 发送请求（如 SENTINEL get-master-addr-by-name <master-name>）来查询当前的主节点地址。
    • 当发生故障转移后，Sentinel 会更新其内部状态。客户端再次查询时，就能获取到新主节点的地址，从而实现自动重定向。

优点

  1. 高可用（HA）： 自动检测主节点故障并完成故障转移，大大减少服务中断时间（通常在秒级）。
  2. 自动化： 无需人工干预主节点故障恢复过程。
  3. 官方支持 & 成熟稳定： Redis 官方原生支持，经过大量生产环境验证。
  4. 扩展性： 读能力可以通过添加从节点轻松扩展。
  5. 数据冗余： 数据在多个节点（主+从）上有副本。

缺点 / 注意事项

  1. 写能力无法扩展： 仍然只有一个主节点负责所有写操作。写性能瓶颈取决于单台主节点的能力。这是哨兵模式最主要的限制。
  2. 故障转移期间的短暂不可用/数据丢失：
    • 不可用： 从主节点故障被确认到选出并提升新主节点，再到客户端感知到新配置，期间写服务是中断的。部分读服务也可能受影响（取决于客户端库实现）。
    • 异步复制导致数据丢失： 故障转移时，如果原主节点有部分数据还没来得及复制到被提升的新主节点，这些数据就会永久丢失。
  3. 配置管理： 客户端需要配置 Sentinel 节点的地址列表，并能正确处理 Sentinel 返回的配置变更。
  4. 资源消耗： 运行多个 Sentinel 节点需要额外的服务器资源。
  5. 脑裂问题（Split-Brain）：
    • 场景： 网络分区导致主节点和部分 Sentinel/客户端/从节点分隔在不同的网络区域。
    • 风险： 被隔离的原主节点可能还在接受写操作（如果客户端也在这个分区内），而另一个分区已经选举出了新主节点也在接受写操作。网络恢复后，两个“主节点”的数据将发生严重冲突。
    • 缓解措施：
      • min-slaves-to-write / min-replicas-to-write：主节点只有在至少有 N 个从节点连接时才能接受写操作。如果网络分区导致主节点连接的从节点少于 N，它会停止写入。
      • min-slaves-max-lag / min-replicas-max-lag：主节点只接受连接时间小于指定延迟的从节点计入 min-slaves-to-write。确保复制延迟过大的从节点不计入，保证数据一致性。
      • 合理的部署： 将 Sentinel 和 Redis 节点部署在可靠的网络环境中，避免单点网络故障。将 Sentinel 部署在独立的、可靠的物理机/VM上。

三、 其他高可用架构（对比与适用场景）

  1. Redis Cluster：
    • 核心： 官方提供的分布式数据分片（Sharding）方案，同时实现了高可用和水平扩展（读写能力）。
    • 结构： 数据自动分片到多个主节点（通常16384个槽）。每个主节点有1-N个从节点（副本）。
    • 高可用： 每个分片（主+从）内部就是一个类似哨兵的单元（由集群自身管理故障转移，无需额外Sentinel进程）。主节点故障时，其从节点会自动提升。
    • 优点： 读写均可水平扩展，无中心节点，自动分片和故障转移。
    • 缺点： 配置和管理更复杂，客户端需要支持集群协议（重定向 MOVED/ASK），某些命令/事务/多键操作受限（键必须在同一槽位），迁移和扩容相对复杂。
    • 适用场景： 数据量超大、读写并发极高、需要水平扩展能力的场景。
  2. 基于代理的分片（如 Twemproxy, Codis）：
    • 核心： 在客户端和 Redis 节点之间引入一个代理层。代理负责接收请求，根据分片规则（如一致性哈希）将请求路由到后端的 Redis 实例（通常是主从+哨兵构成的多个独立分片）。
    • 高可用： 后端每个分片内部使用哨兵模式保证高可用。代理需要能感知后端分片的主节点变化。
    • 优点： 对客户端透明（客户端像连接单节点一样连接代理），可以水平扩展（读写），分片策略灵活。
    • 缺点： 增加代理层带来额外延迟和单点/性能瓶颈（代理本身需要高可用部署），运维复杂度增加（代理+后端集群），多跳网络开销。
    • 适用场景： 在 Redis Cluster 成熟之前较流行，或需要特定分片策略/客户端兼容性时。现在通常更推荐 Redis Cluster。

总结与选择建议

  • 哨兵模式（Sentinel）是最常用、最成熟的 Redis 高可用方案。 它解决了主从复制中主节点故障需要人工干预的核心问题，自动化程度高，配置相对简单，特别适合读多写少、数据量不是特别巨大（单主节点能承载）、需要高可用保障的场景。
  • 选择哨兵模式时：
    • 确保部署至少3个（奇数个） Sentinel 节点（跨机架/可用区部署更佳）。
    • 仔细配置 down-after-milliseconds, parallel-syncs, failover-timeout, quorum 等关键参数以适应你的网络环境和容错需求。
    • 务必配置 min-slaves-to-write 和 min-slaves-max-lag 来缓解脑裂风险。
    • 使用支持 Sentinel 的客户端库，并正确实现故障转移后的重连逻辑。
  • 如果面临写性能瓶颈或数据量远超单节点容量： 应优先考虑 Redis Cluster。
  • 如果需要特定的分片策略或对客户端完全透明，且能接受代理层的开销： 可以考虑 基于代理的分片（如 Codis），但 Redis Cluster 通常是更原生的选择。

简单决策树：

  1. 需要高可用 + 单主节点能处理所有写负载 + 数据量在单节点容量内？ -> 哨兵模式 (首选)
  2. 需要高可用 + 单主节点写性能/容量不够？ -> Redis Cluster
  3. 需要高可用 + 特定分片需求/客户端兼容性要求？ -> 基于代理的分片 (Codis等) (次选，优先评估 Redis Cluster 是否满足)

哨兵模式因其成熟度、相对简单性和对大多数场景的适用性，无疑是 Redis 实现高可用的基石和最广泛采用的架构。
